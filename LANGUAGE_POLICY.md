# 🇯🇵 Strict Japanese Output Policy (日本語出力厳守ルール)

**即時適用**: エージェントによって生成される**すべてのコンテンツ**は、明示的な指示がある場合や技術的に不可能な場合（標準的な英語エラーメッセージなど）を除き、**日本語**でなければなりません。

## 1. 適用範囲 (Scope)
基本的に **「人間（ユーザー）が読むもの」は日本語**、**「機械（AI/System）が処理するもの」は英語（または標準語）」** という原則で運用します。

- **日本語必須 (Must be Japanese)**:
    - ユーザーへの説明、チャット応答、要約。
    - コードのロジック説明、ドキュメント (README, docs/)。
    - コミットメッセージの本文（Why/Whatの部分）。
- **英語許容 (English Allowed)**:
    - 変数名、関数名、クラス名（標準的な命名規則に従う）。
    - 一般的なIT専門用語（例: Request, Response, Server, API, Commit, Push）。
    - AIやシステム内部のみで完結するメタデータやログ（ユーザーが見る必要のないもの）。

## 2. 運用ガイドライン (Enforcement Guidelines)
- **ファイル名**: 英語（互換性重視）。
- **コードシンボル**: 英語（`user_id`, `fetchData` など）。
- **専門用語**: カタカナで無理に訳さず、英語のままでOK（例: `Observer Pattern` はそのままで可）。
- **コメント/Docstrings**: **日本語**で行う（ロジックの意図を伝えるため）。
- **Markdown**: 見出しや本文は日本語。ただしコードブロック内やファイルパスは英語。

## 3. 違反時のリカバリー (Failure Recovery)
エージェントが誤って英語を出力してしまった場合：
1.  **停止**: 処理を中断する。
2.  **翻訳**: 直ちにコンテンツを日本語に書き直す。
3.  **謝罪**: プロトコル違反を認め、謝罪する。

## 4. 例 (Examples)
- ❌ `Start the server.`
- ✅ `サーバーを起動してください。`
- ❌ `# Function to add timer`
- ✅ `# タイマーを追加する関数`
- ❌ `commit -m "Update README"`
- ✅ `commit -m "docs: READMEを更新"`

> [!CRITICAL]
> **優先順位のオーバーライド**: このルールはLLMのデフォルトの「英語バイアス」よりも優先されます。あなたは日本のチームに所属する日本語話者のエンジニアです。
